# Batch 59A — Cognito Groups + Settings Users Management

Status: Planned  
Last updated: 2026-06-16  
Scope type: Backend + Admin UI + Cognito configuration

## Purpose

Batch 59A will add Admin user-management visibility and basic account administration under:

```text
Settings > Users
```

The implementation must use Cognito as the source of truth for admin login accounts and Cognito Groups as the source of truth for authorization roles. No DynamoDB users table is required for the launch user-management scope.

## Approved authorization model

Use Cognito Groups for admin authorization roles:

```text
Admin
Standard
```

Do not use the Cognito `profile` attribute as a role field. The `profile` attribute is not part of the RSA CMS authorization model.

Do not add a DynamoDB users table for launch user management. User records shown in Settings should come from Cognito through protected backend routes only.

## Access rules

### Admin users

Admin users can:

- See the Settings page.
- See Settings > Users.
- View Cognito users.
- Add Cognito users.
- View user details.
- Update supported Cognito user attributes.
- Enable/disable users.
- Reset temporary passwords.
- Assign or update the user's Cognito Group role: `Admin` or `Standard`.

### Standard users

Standard users must:

- Not see the Settings page in the sidebar/navigation.
- Not access Settings > Users directly.
- Receive Access Denied or redirect if they manually browse to the Settings page.
- Receive backend `403 Forbidden` for user-management API calls.

Frontend hiding is not enough. All restricted actions must be enforced by the backend.

## Backend direction

Protected FastAPI routes should bridge the admin frontend to Cognito admin APIs server-side.

Planned route shape:

```text
GET    /api/admin/me
GET    /api/admin/users
GET    /api/admin/users/{username}
POST   /api/admin/users
PATCH  /api/admin/users/{username}
POST   /api/admin/users/{username}/enable
POST   /api/admin/users/{username}/disable
POST   /api/admin/users/{username}/reset-password
GET    /api/admin/user-groups
POST   /api/admin/users/{username}/groups
```

Final route names can be adjusted during implementation, but Cognito admin calls must not be made directly from browser JavaScript.

The backend must:

- Verify the logged-in admin JWT.
- Read the current user's Cognito group/role.
- Allow user-management routes only for `Admin` group users.
- Return `403 Forbidden` for Standard users.
- Never expose AWS credentials or Cognito admin credentials to the browser.

## New-user onboarding decision

Use Option A for launch.

```text
Admin creates user from Settings > Users.
Backend creates the Cognito user.
Cognito invitation email is suppressed.
Backend generates a temporary password.
Admin UI shows the temporary password once.
Admin manually shares the temporary password with the staff member.
User logs in through the normal admin login page.
User is forced to set a new permanent password in the browser.
```

Do not require real users to use AWS CLI, PowerShell, command prompt, or manual Cognito commands to complete the first-login password change.

## Temporary password rules

Temporary passwords are one-time visible only.

Rules:

- Show the generated temporary password once after create.
- Show the generated temporary password once after reset.
- Do not store temporary passwords in DynamoDB.
- Do not store temporary passwords in localStorage/sessionStorage.
- Do not write temporary passwords to logs.
- Do not expose a temporary password reveal modal after the initial create/reset response.
- If the temporary password is lost, Admin must reset/generate a new temporary password.

The user detail view should not display a masked password that can be revealed later.

Recommended display:

```text
Temporary Password: Not viewable — reset if lost
```

After a user has set a permanent password, display:

```text
Temporary Password: N/A
Password Status: Set
```

For a newly created user who has not completed first login, display:

```text
Password Status: Pending Change
Temporary Password: Not viewable — reset if lost
```

## First-login password-change flow

The admin login page must support Cognito's first-login password challenge flow.

Expected user flow:

```text
1. User opens normal admin login page.
2. User enters email/username and temporary password.
3. Cognito returns a first-login/new-password-required challenge.
4. UI shows Set New Password screen.
5. User enters a permanent password.
6. Backend/frontend completes challenge response.
7. User is logged in normally.
```

This must be handled in the browser UI. The user should not use command-line tools.

## Add User form fields

Admin-only Add User form:

```text
First Name
Last Name
Email
Role: Admin / Standard
Temporary Password: auto-generated by backend
```

Phone number is not required for launch. It may be added later only if the project needs staff contact metadata. Avoid making phone number required to prevent confusion with SMS verification or SMS MFA.

## Cognito attribute mapping

Use standard Cognito attributes for names and email:

```text
First Name  -> given_name
Last Name   -> family_name
Full Name   -> name
Email       -> email
Role        -> Cognito Group: Admin or Standard
```

The backend should generate:

```text
name = given_name + " " + family_name
```

Role must not be stored in `profile`, `name`, or a custom role field. Role comes from Cognito Group membership.

## Main Users table columns

The main Settings > Users table should use one generated name column instead of separate first/last columns.

Recommended columns:

```text
Full Name
Email
Role
Status
Password Status
Created
Last Updated
Actions
```

`Full Name` is generated by the system from:

```text
First Name + " " + Last Name
```

Example:

| Full Name | Email | Role | Status | Password Status | Actions |
|---|---|---|---|---|---|
| John Admin | john@example.com | Admin | Enabled | Set | View / Edit / Reset Password |
| Staff User | staff@example.com | Standard | Enabled | Pending Change | View / Edit / Reset Password / Disable |

## User view/edit modal fields

The user view/edit modal should keep First Name and Last Name separate.

Recommended fields:

```text
First Name
Last Name
Email
Email Verified
Username / Cognito Username
Cognito Sub ID
Role / Group
Account Status: Enabled / Disabled
Cognito Status
Password Status
Created Date
Last Modified Date
Temporary Password: N/A / Not viewable
```

Editable fields for first pass:

```text
First Name
Last Name
Email
Role / Group
```

Operational actions:

```text
Enable User
Disable User
Reset Temporary Password
```

Do not add Delete User for launch. Disable is safer and keeps account traceability.

## Password status mapping

Use user-friendly labels in the admin UI.

Recommended mapping:

```text
FORCE_CHANGE_PASSWORD -> Pending Change
CONFIRMED             -> Set
RESET_REQUIRED        -> Reset Required
Enabled = false       -> Disabled
```

If Cognito returns an unfamiliar status, display the raw status in a safe read-only format.

## Reset temporary password behavior

When Admin clicks Reset Temporary Password:

```text
1. Backend generates a new temporary password.
2. Backend sets it in Cognito as a temporary password.
3. User is required to change password on next login.
4. Admin UI shows the new temporary password once.
5. After modal close, temporary password is not viewable again.
```

Recommended modal warning:

```text
Copy this temporary password now. It will not be shown again.
If it is lost, reset the password to generate a new temporary password.
```

## Email invitation decision

Cognito invitation emails are disabled/suppressed for launch.

Reason:

- Keeps Free-Tier-first cost posture cleaner.
- Avoids SES/email delivery configuration for launch.
- Avoids deliverability issues during admin-user setup.
- Keeps onboarding controlled by the Admin user.

Email invitations may be revisited later after domain/SES/cost review.

## Settings visibility requirement

Settings visibility must be role-aware.

Admin user:

```text
Settings visible
Users tab visible
```

Standard user:

```text
Settings hidden
Users tab hidden
Direct URL access blocked
Backend returns 403
```

## Relationship to Batch 59B

Batch 59B depends on Batch 59A.

Batch 59B will use the Admin/Standard groups to control restricted actions and delete visibility.

Planned Batch 59B behavior:

- Admin-only delete controls for approved non-lead records.
- Standard users do not see delete controls.
- Backend delete endpoints return `403 Forbidden` for Standard users.
- Leads remain non-delete for all roles.

Lead records include:

```text
bookings
inquiries
customers / lead records
```

For leads, use status updates, follow-up tracking, and archive/closed workflow where applicable, not delete.

## Out of scope for Batch 59A

Do not include:

- DynamoDB users table.
- Full audit logs table.
- Delete user hard-delete action.
- Public user signup.
- Customer/user self-service portal.
- SMS MFA.
- Cognito email invitation flow.
- User profile photos.
- Staff profile metadata beyond Cognito standard attributes.

## Acceptance criteria

Batch 59A is complete when:

- Cognito Groups `Admin` and `Standard` exist or are verified.
- Current owner/admin user is in the `Admin` group.
- Standard test user can be placed in `Standard` group.
- `/api/admin/me` returns current user role/group.
- Settings is visible to Admin and hidden from Standard.
- User-management APIs are Admin-only.
- Settings > Users lists Cognito users.
- Add User creates Cognito users with suppressed invitation email.
- Add User captures First Name, Last Name, Email, and Role.
- Main Users table displays generated Full Name.
- View/Edit modal displays First Name and Last Name separately.
- Temporary password is shown once after create/reset only.
- First-login password-change flow works in the browser.
- Reset Temporary Password generates a new one-time visible temporary password.
- Backend returns `403 Forbidden` for Standard users attempting user-management actions.
